# chobitfile サービス構想メモ

最終更新: 2026-07-21

技術選定の前提となるプロダクト理解をまとめる。実装仕様書ではなく、検討時点のメモである。

第一弾の確定仕様は [MVP 仕様](./mvp-spec.md) を正本とする。

## 概要

指定したサイズ・形式のダミーファイル（PDF、CSV、画像など）を生成するツール。

想定ユースケース:

- ファイルアップロード機能のテストで「ちょうど 10MB の PDF」などが欲しい
- 形式バリデーション・サイズ上限・プレビューなどの確認

## 形式ごとの難易度

パディング領域の有無・構造制約で難易度が決まる。

| 難易度 | 形式 | ポイント |
|---|---|---|
| 易 | CSV / テキスト、生バイナリ | 制約なくバイト数調整可能 |
| 易〜中 | JSON | パースを壊さない範囲で調整 |
| 中 | PDF | コメント行・未使用ストリームにパディング |
| 中 | PNG / JPEG | 未知チャンク・APPn セグメントにパディング。CRC32 計算が必要 |
| やや難 | DOCX / XLSX / PPTX（ZIP 系） | ダミーファイル追加 + ZIP ヘッダオーバーヘッド計算 |
| 難 | MP4 / MP3 | free/skip atom、ID3v2 パディング。構造組み立てのコストが高い |

## 実行場所

**サーバーではなくクライアント（ブラウザ）側生成**を前提とする。

理由:

- 帯域コストがゼロ（生成結果をネットワーク転送しない）
- ストレージコストがゼロ（サーバーにダミーを置かない）
- プライバシー上もアップロード不要

実装候補（検討時点）:

1. 素の JS（メインスレッド）— テキスト系・生バイナリ向け
2. Web Workers — PDF / PNG / JPEG / ZIP 系で UI ブロックを避ける
3. WASM — CRC32 や ZIP、MP4 atom などを厳密に扱う場合に有利になり得る、という仮説

大きいファイルは Blob のチャンク結合、または File System Access API でメモリ確保を抑える方針も候補。

技術選定の検証結果は [JS vs MoonBit WASM](./tech-selection-js-vs-moonbit-wasm.md) を参照。
