# 技術選定: JS vs MoonBit (WASM)

| 項目 | 内容 |
|---|---|
| 日付 | 2026-07-21 |
| 対象 | ブラウザ側ダミーファイル生成の実装言語 |
| 比較 | 素の JavaScript / MoonBit → wasm-gc |
| 結論 | **MVP は素の JS（必要なら Web Worker）。MoonBit/WASM は前提にしない** |
| 再現 | ベンチ実装（`bench/` / `moonbit-bench/`）は採用判断後に削除済み。数値は本ドキュメントに残す |

関連: [サービス構想メモ](./service-concept.md)

---

## 1. 検証の目的

「WASM の方が速そう」を決め打ちせず、chobitfile で実際に使う処理に近いカテゴリで **JS と MoonBit WASM を比較**する。

併せて、性能以外（シンプルさ・可読性・保守性）も評価し、プロダクト方針を決める。

### 検証しないこと / スコープ外

- Rust / C / AssemblyScript など MoonBit 以外の WASM 言語
- サーバ生成との比較
- 実ブラウザ UI の体感（本計測の主結果は Node.js）
- WASM 生成結果を `Uint8Array` / `Blob` にコピーして返す経路のオーバーヘッド  
  （本ベンチは各実装内で計算完結。コピーを入れると WASM 側にさらに不利に働く可能性が高い）

---

## 2. ベンチマーク設計

### 2.1 処理カテゴリ

パフォーマンス差が出やすいのは単純ゼロ埋めより **計算を伴う処理**。次の 5 つに分解した。

| キー | 内容 | 想定ドメイン |
|---|---|---|
| `pad_zeros` | N バイトのゼロ埋め | 生バイナリ・単純パディング |
| `pad_pattern` | インデックス依存パターン書き込み | ランダム風パディング |
| `crc32_pattern` | 定値埋め + IEEE CRC-32（テーブル方式） | PNG 等 |
| `zip_store` | store（無圧縮）ZIP の組み立て | DOCX / XLSX / PPTX |
| `text_pad` | PDF 風コメントパディング（文字列組み立て） | PDF / テキスト系 |

JS と MoonBit で **同一アルゴリズム・同一入出力** になるよう実装し、毎回返り値一致を検証した。

### 2.2 計測条件

| 項目 | 内容 |
|---|---|
| サイズ | 1MB, 10MB, 100MB |
| 計時 | `performance.now()`（生成開始〜処理完了） |
| 統計 | ウォームアップ後の **中央値 (ms)** |
| ウォームアップ / 反復 | 2 / 5（フル計測時） |
| 環境 | Node.js v26.3.0 / darwin / `--expose-gc` |
| MoonBit | `wasm-gc` + JS String Builtins、release ビルド |
| WASM 成果物サイズ | 約 6.4 KB |

### 2.3 コード配置

ベンチ実装（`bench/` / `moonbit-bench/`）は採用判断後に削除した。結果の数値のみ本ドキュメントに残す。

## 3. 性能結果

### 3.1 数値（中央値 ms）

| 処理 | サイズ | JS (ms) | WASM (ms) | WASM/JS | 勝者 |
|---|---:|---:|---:|---:|---|
| 単純ゼロ埋め | 1MB | 0.014 | 0.345 | 24.6x | js |
| パターン埋め | 1MB | 0.995 | 0.670 | 0.67x | wasm |
| CRC32（埋め+計算） | 1MB | 2.683 | 3.962 | 1.48x | js |
| ZIP store 構築 | 1MB | 2.738 | 4.694 | 1.71x | js |
| PDF風テキスト組み立て | 1MB | 0.195 | 1.820 | 9.33x | js |
| 単純ゼロ埋め | 10MB | 0.124 | 3.283 | 26.5x | js |
| パターン埋め | 10MB | 7.901 | 6.436 | 0.81x | wasm |
| CRC32（埋め+計算） | 10MB | 26.843 | 39.904 | 1.49x | js |
| ZIP store 構築 | 10MB | 27.156 | 48.492 | 1.79x | js |
| PDF風テキスト組み立て | 10MB | 1.901 | 19.131 | 10.1x | js |
| 単純ゼロ埋め | 100MB | 1.215 | 40.154 | 33.0x | js |
| パターン埋め | 100MB | 64.440 | 72.848 | 1.13x | js |
| CRC32（埋め+計算） | 100MB | 266.259 | 411.109 | 1.54x | js |
| ZIP store 構築 | 100MB | 270.201 | 500.119 | 1.85x | js |
| PDF風テキスト組み立て | 100MB | 27.884 | 202.945 | 7.28x | js |

- **WASM/JS**: 1 未満なら WASM が速い、1 超なら JS が速い
- **勝者**: 10% 以上の差があるときのみ付与（それ以外は表では優勢側を記載）

### 3.2 カテゴリ別サマリ

| 処理 | 傾向 | 所見 |
|---|---|---|
| 単純ゼロ埋め | **JS が 25–33 倍速い** | `new Uint8Array(n)` のゼロ初期化がエンジン最適化済み |
| パターン埋め | ほぼ互角（〜1.2x 前後） | ループ書き込み。サイズで勝者入れ替わり |
| CRC32 | **JS が約 1.5 倍速い** | テーブル CRC でも V8 の TypedArray ループが強い |
| ZIP store | **JS が約 1.7–1.9 倍速い** | CRC + バッファ組み立てで WASM 側コストが乗る |
| PDF 風テキスト | **JS が約 7–10 倍速い** | 文字列操作は JS エンジンが得意 |

### 3.3 「WASM が速い」定説とのずれ

検証前の仮説では、CRC32 のようなビット演算主体は WASM 有利になりやすいと見ていた。  
実測では **MoonBit wasm-gc は当該ワークロードで JS を上回らなかった**。

考えられる要因:

1. 比較対象が Rust 製の線形メモリ WASM ではなく **MoonBit wasm-gc**（GC 配列・境界チェックあり）
2. ダミー生成のホットパスは「巨大バッファ確保 + 単純ループ」で、**V8 の TypedArray がすでに十分速い**
3. 本ベンチは **JS↔WASM メモリコピーを含まない**（各実装内完結）。実プロダクトで WASM 結果を `Blob` にする場合、コピーがさらに乗る
4. ウォームアップ後の JS は JIT が効く。初回だけの比較では WASM 有利に見えやすい

### 3.4 再現手順

ベンチ用コードは削除済みのため、現行リポジトリでは再実行できない。当時の条件は §2.2、結果は §3.1 を参照。

## 4. 性能以外: シンプルさ・可読性・保守性

性能に加え、コード品質の観点でも比較した（2026-07-21）。

### 4.1 総合

| 観点 | 勝者 | このプロダクト文脈での理由 |
|---|---|---|
| シンプルさ | **JS** | ブラウザがそのまま動く。ビルド・FFI・型変換が不要 |
| 可読性 | 互角〜**JS 微有利** | バイナリ操作本体は互角。Web API まで含めると JS が一本 |
| 保守性 | **JS** | エコシステム・人材・ツール・デバッグが揃う |
| 型安全性・境界の厳密さ | **MoonBit** | コンパイル時に多くのミスを潰せる |

### 4.2 シンプルさ

chobitfile の最終形は次の一本線になる。

入力 UI → バイト列生成 → `Blob` / ダウンロード（必要なら Worker・File System Access API）

**JS** ならこの世界に閉じる。

**MoonBit** を入れると生成ロジック以外に必ず増える。

- `moon build` と成果物配布
- `WebAssembly.instantiate`（JS String Builtins 等の設定）
- `Int` / `UInt` / `UInt16` の変換や `reinterpret_as_*`
- wasm-gc ではバイト列を `Uint8Array` として安価に返す経路が素直でない

「PDF にコメントでパディングして Blob にする」規模に対して、二言語境界のコストが大きい。

### 4.3 可読性

- **アルゴリズム本体**（CRC32・ZIP ヘッダ）: 互角。MoonBit の `Buffer.write_uint_le` は JS の `DataView` + オフセット手計算より意図が読みやすい場面もある
- **ドメイン全体**（UI・ダウンロード・Worker）: JS の方がストーリーが一本で読める
- MoonBit は「中核は読みやすいが、境界で意図が割れやすい」

### 4.4 保守性

JS が有利な点:

- ブラウザ生成ツールの事例・ライブラリが圧倒的
- DevTools でそのままデバッグできる
- Biome / テストランナー / Playwright など既存スタックと直結
- 形式追加が「関数を足す」で済む

MoonBit が重い点:

- 言語・stdlib・FFI の変化がまだ速い
- WASM 障害時の切り分けコスト
- バイナリ仕様の参照実装は JS / Python / Rust が中心
- 性能で勝っていないのにツールチェイン依存だけ残る

MoonBit が有利な点（限定的）:

- 静的型で形式ごとの生成器を厳密にモデル化しやすい
- 純粋なコアのテスト容易性

ただし本質的なバグは「ZIP オフセット計算ミス」のような **仕様理解不足** が多く、型だけでは防げない。

### 4.5 MoonBit が保守面で勝つ条件（今回は非該当）

- 同じ生成ロジックを CLI / サーバ / 他ホストでも共有したい
- コアが十分大きく、型とテストで長期保守したい
- チームが MoonBit に慣れている

chobitfile はブラウザ特化・形式追加が主・性能も JS 優勢のため、条件に当たりにくい。

## 5. 決定

### 5.1 採用

1. **本体は JS（または TypeScript）**  
   - `Uint8Array` / `Blob` / 必要なら File System Access API
2. **重い形式は Web Worker**  
   - UI ブロック回避。WASM の有無とは独立
3. 形式は難易度の低いものから縦スライス  
   - CSV / 生バイナリ → JSON → PDF → PNG/JPEG → ZIP 系 → メディア

### 5.2 不採用 / 後回し

| 案 | 理由 |
|---|---|
| 全面 MoonBit/WASM 実装 | 性能・統合コスト・保守性のいずれでも優位が出ない |
| MVP からのハイブリッド（コア WASM） | 境界コストに見合う速度差がない |
| 線形メモリ backend でのバイト返却検証 | 計算内核で既に JS 優位のため優先度低 |

### 5.3 再検討トリガ

次のいずれかが起きたら WASM / MoonBit を再評価する。

- 実測で JS がボトルネックになり、かつ計算律速と特定できた
- ブラウザ以外のホストとロジック共有が必須になった
- より有利な WASM 言語・線形メモリ経路で再ベンチした結果、明確な差が出た

## 6. プロダクトへの落とし込み（実装時の指針）

- 生成器は形式ごとにモジュール分割する（例: `generators/csv.ts`, `generators/pdf.ts`）
- 共有ユーティリティ（`crc32`, `writeU32LE` 等）は薄いヘルパにまとめる  
  - MoonBit `Buffer` の読みやすさは JS ヘルパで再現可能
- 型が欲しければ **TypeScript** で足りる。MoonBit まで踏み込む必要は薄い
- 100MB 級はメインスレッドを避け、Worker + チャンク / ストリームを検討する
